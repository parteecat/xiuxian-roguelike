# AI 修仙人生模拟器重构文档 v2

## 0. 文档定位

本版本用于将现有项目从：

- AI 驱动的高自由度修仙互动叙事
- 聊天 + 选项推进的轻剧情游戏
- 偏“小说生成器”的修仙体验

重构为：

> **一个以阶段推进为骨架、以因果回收为特色的 AI 修仙人生模拟器。**

这份文档优先服务于 **AI 重构与编码执行**，因此强调：

- 可落地的玩法骨架
- 可实现的系统边界
- 前端本地规则层与 LLM 叙事层的职责拆分
- MVP 范围控制
- 数据结构与模块拆分建议

---

## 1. 重构目标

### 1.1 产品目标

玩家体验不再是：

- 看 AI 继续写修仙故事
- 通过聊天随意尝试修仙行为
- 在缺少强目标的世界里自由游荡

而是：

- 在有限寿元、阶段目标、大劫倒计时下规划一世修仙人生
- 在每个阶段做出有代价的关键抉择
- 被过去的行为在未来以“因果”形式重新找上门
- 最终收获一份只属于自己的修仙人生结算与传记

### 1.2 要解决的问题

当前版本核心问题：

1. 更像互动小说，不像游戏
2. 有世界观和文本，但缺少持续推进的动力
3. 行动有内容差异，但缺少策略差异
4. NPC 可交互，但长期记忆和命运回响弱
5. 主线压力不足，玩家没有“必须继续玩”的理由
6. AI 很会写，但系统后果不足，导致不够上瘾

### 1.3 v2 的核心改造方向

v2 不再以“自由聊天”为核心，而是转为：

- **阶段推进**：人生被拆成几个有明确目标的阶段
- **有限行动**：每个时间单位只能做有限事情
- **主线压力**：存在固定或半随机的大事件倒计时
- **因果回收**：过去关键行为会在未来被回收、奖惩或反噬
- **人生结算**：每局结束都有结局标签 + AI 小传

---

## 2. 产品一句话定义

> 在寿元、大劫与人情因果交织的修仙世界中，玩家以阶段推进的方式经营自己的一生；每一次选择，都可能在未来变成机缘、代价、报应或救赎。

---

## 3. 设计原则

## 3.1 游戏性优先于叙事性

AI 的文本能力只作为表现层加成。

真正的核心应是：

- 状态变化
- 风险收益权衡
- 时间推进
- 因果后果
- 人生路线分化

禁止再次滑回“长篇剧情 + 弱选择”。

## 3.2 阶段推进优先于自由输入

自由输入保留，但不再是主要交互方式。

主要交互方式应为：

- 系统给出高质量结构化行动
- 玩家进行有限选择
- 系统和 AI 共同结算

自由输入仅作为补充入口。

## 3.3 选择必须有代价

重要行动必须绑定真实成本，例如：

- 时间
- 寿元
- 灵石
- 气血 / 真气
- 名声
- 心境 / 心魔
- NPC 信任
- 错过别的机会

## 3.4 世界必须自动推进

不是只有玩家在行动。

世界应持续变化：

- 倒计时减少
- 大事件逼近
- NPC 立场变化
- 阵营态势变化
- 地点状态变化
- 旧因果开始回收

## 3.5 AI 负责“命运质感”，前端负责“规则真相”

前端本地规则层负责：

- 状态真值
- 概率计算
- 条件判断
- 资源变化
- 事件推进
- 死亡 / 结局 / 解锁

LLM 负责：

- 当前局势摘要
- 风险提示语言包装
- 行动文案润色
- NPC 个性化反应
- 因果回收演出
- 结局小传生成

---

## 4. 目标体验

如果 v2 成功，玩家在前 10~20 分钟内应感受到：

1. 我不是在看故事，我是在经营这一世的人生
2. 当前每回合都有明确的近目标和压力
3. 我必须在成长、关系、资源、风险之间做取舍
4. 过去做过的小事会在后面回来影响我
5. 每一局的人生路线都能明显不同
6. 死亡或结局后，我会想立刻再开一局试另一条命

---

## 5. 产品定位：不是开放 RPG，而是修仙人生模拟器

项目定位调整为：

> **以人生模拟骨架驱动的修仙命运模拟游戏**

它借鉴的不是开放世界修仙 RPG，而是：

- 阶段推进式成长模拟
- 一局一世的命运经营
- 有限时间与有限行动的资源规划
- 多结局、多重开、多人生路线

与一般“人生模拟”不同的核心差异是：

1. 修仙题材下的寿元压力与境界门槛
2. 修仙世界特有的因果、人情、秘境、大劫、心魔
3. AI 驱动的动态因果回收与个性化人生传记

---

## 6. 核心循环

## 6.1 单月循环（推荐作为 MVP 基础）

推荐以“月”为回合最小单位。

每月流程：

1. 结算世界自动推进
2. 刷新当前阶段压力与倒计时
3. 检查是否触发强制事件 / 因果回收 / 主线节点
4. 系统生成当月可选行动（2~3 个行动槽）
5. 玩家依次选择行动
6. 前端本地结算数值与状态变化
7. AI 输出结果摘要、关系反馈、事件氛围文本
8. 月末结算：资源变化、关系变化、因果变化、阶段推进
9. 进入下个月

## 6.2 中期循环（阶段循环）

每个阶段包含：

- 明确目标
- 明确资源瓶颈
- 明确危险
- 明确关键节点事件
- 明确阶段失败成本

阶段结束条件可能是：

- 境界突破成功
- 关键主线节点完成
- 时间进入下一人生阶段
- 阵营站队后人生路径锁定

## 6.3 长期循环（一局一世）

一局游戏 = 一世修仙人生。

基本路径：

- 出身与命格生成
- 少年/入道期求生
- 炼气期立足
- 筑基期定道
- 金丹期应劫
- 终局阶段结算

局终输出：

- 结局标签
- 人生里程碑
- 最重要的三次选择
- 关键因果链
- 关系总结
- AI 修仙小传
- 局外解锁

---

## 7. 阶段结构设计

## 7.1 阶段总览

MVP 推荐先做三段结构：

1. **入道/炼气前期**
2. **炼气后期/筑基期**
3. **终局结算段**

正式版可扩展为：

1. 少年期
2. 炼气期
3. 筑基期
4. 金丹期
5. 元婴/终局期

## 7.2 阶段一：入道 / 炼气前期

### 阶段目标

- 活下来
- 获得稳定资源来源
- 确定早期方向
- 结识第一批关键人物

### 阶段压力

- 没背景
- 没功法
- 灵石紧张
- 修为低，容错低
- 身份未定（散修 / 宗门 / 家族附庸 / 黑市边缘）

### 关键问题

- 先赚钱还是先修炼
- 进宗门还是走散修路线
- 信谁、帮谁、得罪谁
- 要不要赌第一波机缘

### 阶段关键词

- 生存
- 机会
- 投资
- 第一因果

## 7.3 阶段二：炼气后期 / 筑基期

### 阶段目标

- 完成第一次身份跃迁
- 确定道途 / build 方向
- 获得筑基条件
- 形成稳定人脉与敌对关系

### 阶段压力

- 筑基材料
- 功法路线选择
- 阵营牵引
- 心魔与代价开始显现
- 旧行为开始被回收

### 关键问题

- 稳扎稳打还是赌命突破
- 站队还是保持中立
- 为资源牺牲什么
- 谁值得长期投资

### 阶段关键词

- 立道
- 站队
- 代价
- 第一次大因果回收

## 7.4 阶段三：金丹 / 终局段（MVP 可简化）

### 阶段目标

- 面对世界级冲突或终局压力
- 回收前期因果
- 决定你成为什么样的人

### 阶段压力

- 大劫/大战/秘境/妖潮
- 人脉兑现或反噬
- 旧仇复燃
- 重大抉择的终局收口

### 关键问题

- 飞升 / 守护 / 归隐 / 成魔 / 复仇 / 牺牲
- 保全自己还是承担更大的代价
- 是否为过去行为买单

### 阶段关键词

- 应劫
- 回响
- 终局自我定义

## 8. 主线压力系统

这是 v2 的核心新增系统。

不是写死一条线性主线剧情，而是为每一局生成一个 **主线压力源**。

## 8.1 主线压力定义

主线压力 = 一个会持续推进、具有明确倒计时、会影响大量行动价值的中长期事件。

## 8.2 MVP 推荐主线压力类型

### A. 秘境开启线

- 第 X 年 / X 月秘境开启
- 玩家需在此之前准备修为、情报、队友、法器、资格
- 错过则失去一次重大跃迁机会

### B. 宗门大战线

- 某时间点宗门冲突爆发
- 玩家要提前决定立场、资源和人际归属
- 战前站队决定战后命运

### C. 妖潮边关线

- 妖潮逼近，地图区域危险度上升
- 玩家可守城、逐利、逃遁、借势
- 多条路线对应不同结局和名声

### D. 飞升残卷线

- 某上古遗迹将在特定阶段现世
- 玩家需长期收集碎片 / 情报 / 结盟对象
- 前期小选择会影响最终资格与风险

## 8.3 主线压力的功能

主线压力必须带来：

- 明确的倒计时
- 长线准备动机
- 阶段目标聚焦
- 错过成本
- 世界变化方向

没有主线压力，游戏仍会回到松散叙事。

---

## 9. 因果回收系统（v2 核心特色）

这是本次重构最重要的差异化系统。

## 9.1 设计目标

让玩家感受到：

> 我以前做过的事，没有消失，只是还没回来。

## 9.2 因果的来源

下列行为应被记录为潜在因果种子：

- 救助 / 背弃某 NPC
- 交易 / 贪便宜 / 卖出关键物品
- 接受或拒绝某次请求
- 站队、告密、背叛、护短
- 夺取机缘
- 修炼禁术、做出邪道行为
- 在关键时刻选择自保或牺牲

## 9.3 因果数据结构建议

每条关键行为都应落地为一个 `KarmaSeed` 或 `CausalityRecord`。

建议字段：

```ts
export type CausalityRecord = {
  id: string
  title: string
  sourceTurn: number
  sourceStage: string
  category: 'favor' | 'debt' | 'betrayal' | 'promise' | 'loot' | 'enmity' | 'secret'
  ownerNpcId?: string
  ownerFactionId?: string
  relatedItemId?: string
  relatedLocationId?: string
  summary: string
  intensity: number
  hidden: boolean
  status: 'planted' | 'tracking' | 'returned' | 'resolved' | 'failed'
  expectedReturnPhase?: string
  expectedReturnTags?: string[]
  consequences?: string[]
}
```

## 9.4 因果回收规则

### 回收不应随机乱触发

必须结合：

- 当前阶段
- 当前地点
- 当前主线压力
- NPC 是否仍存活 / 在场 / 可用
- 玩家当前境界与状态
- 该因果的强度与标签

### 回收形式

1. **正向回收**：旧恩情兑现，获得帮助、情报、庇护
2. **负向回收**：旧仇复发，遭遇阻击、抹黑、追杀
3. **灰度回收**：对方因自身处境变化，回报与反噬并存
4. **心魔回收**：并非外部事件，而是内部心境/突破时被翻出
5. **主线绑定回收**：因果在主线高潮时集中引爆

## 9.5 因果回收的表现形式

### 普通层

- NPC 态度变化
- 额外选项解锁 / 锁死
- 价格变化
- 新情报出现
- 某行动成功率变化

### 强演出层

- 旧人重逢
- 恩人或仇敌身份反转
- 某件旧物重现
- 大战前收到回报/报应
- 突破时触发心魔事件

## 9.6 因果系统与 AI 的关系

前端本地负责：

- 记录因果种子
- 判断是否到达回收条件
- 结算回收的数值后果

AI 负责：

- 用修仙世界的语言与情绪把它写得有命运感
- 根据 NPC 视角补足反应与对白
- 生成“原来那时埋下的是这个”的回响质感

---

## 10. NPC 系统重构

NPC 不再主要承担“可聊天对象”，而应承担 **玩法节点 + 因果承载者**。

## 10.1 NPC 的新职责

每个关键 NPC 至少承担 2 项以上功能：

- 资源入口
- 情报入口
- 师承 / 道途入口
- 主线节点
- 因果回收容器
- 情绪价值来源
- 危险 / 背叛 / 竞争来源

## 10.2 MVP NPC 数量控制

每局建议先控制为：

- 3 个关键关系 NPC
- 2 个功能 NPC
- 1 个镜像/宿敌型 NPC

不要一开始做大量 NPC。

## 10.3 NPC 建议字段

```ts
export type NpcState = {
  id: string
  name: string
  title: string
  camp: string
  realm: string
  favor: number
  trust: number
  fear: number
  danger: number
  relationStatus: 'neutral' | 'friendly' | 'ally' | 'suspicious' | 'enemy' | 'lover' | 'mentor'
  firstImpression: string
  moralView: string
  playerImpression: string
  sharedHistory: string[]
  karmaLinks: string[]
  functions: Array<'trade' | 'quest' | 'intel' | 'mentor' | 'romance' | 'enemy'>
  hiddenGoals: string[]
  tags: string[]
  flags: string[]
}
```

## 10.4 NPC 长期记忆要求

AI 生成 NPC 互动时，需要能利用：

- 初见印象
- 最近共同经历
- 长期好感与信任变化
- 曾经的承诺 / 欠债 / 伤害
- 玩家的声名与路线倾向

这使 NPC 看起来像在“记住你”，而不是单纯给文本回复。

## 11. 行动系统重构

## 11.1 从自由聊天改为结构化行动

主要交互改成：

- 每月给出 2~3 次行动机会
- 每次展示 3~5 个策略差异明显的行动选项
- 保留自定义行动作为补充

## 11.2 行动分类

每回合行动应尽量从以下类别中组合：

### 1. 稳健型

- 疗伤
- 休整
- 低风险修炼
- 低风险交易
- 维护关系

### 2. 成长型

- 闭关
- 练功
- 炼丹 / 炼器 / 读卷
- 中风险探索
- 获取关键材料

### 3. 赌博型

- 强行突破
- 深入险地
- 截胡机缘
- 潜入敌营
- 设局反杀

### 4. 社交型

- 拜访 NPC
- 打听情报
- 求师 / 求援 / 结盟
- 赴约

### 5. 事件推进型

- 参加拍卖
- 进入秘境
- 接受站队邀请
- 调查旧因果
- 处理倒计时事件

## 11.3 选项设计要求

禁止同义重复选项。

每个选项都应具有：

- 风险等级
- 主要收益
- 主要代价
- 时间消耗
- 倾向标签
- 关联系统（资源 / NPC / 主线 / 因果）

建议结构：

```ts
export type ActionOption = {
  id: string
  kind: 'rest' | 'cultivate' | 'explore' | 'social' | 'trade' | 'breakthrough' | 'investigate' | 'event' | 'custom'
  title: string
  desc: string
  risk: 'low' | 'mid' | 'high'
  timeCostMonths: number
  primaryRewards: string[]
  primaryCosts: string[]
  tags: string[]
  relatedNpcId?: string
  relatedEventId?: string
  relatedCausalityId?: string
}
```

## 12. 资源系统（v2 重点）

v2 不只是“属性面板”，而是“命运资源面板”。

## 12.1 显性资源（必须展示）

- 境界 / 小境界 / 修为进度
- 气血
- 真气
- 寿元
- 年龄
- 灵石
- 心魔
- 声望 / 恶名
- 当前阶段目标
- 主线倒计时

## 12.2 半显性资源（可通过标签/提示展示）

- 道心稳定度
- 阵营倾向
- 关系信用
- 因果债
- 构筑倾向
- 命格偏移

## 12.3 隐性资源（前端可存，前期不全部展示）

- 某 NPC 的真正看法
- 某因果的回收触发进度
- 某阵营对你的隐藏判定
- 某机缘资格值

---

## 13. 道途 / Build 系统

为了让游戏不是单纯人生排班器，玩家需要逐步形成 **修仙路线**。

## 13.1 MVP 推荐路线标签

- 剑修
- 体修
- 法修
- 丹修
- 社交流 / 魅修
- 邪修
- 苟道 / 生存流

## 13.2 路线实现方式

不做职业锁死，而使用“标签累计”。

玩家的路线标签来自：

- 初始命格
- 功法
- 关键选择
- NPC 师承
- 常用行动
- 因果处理方式

## 13.3 路线标签的作用

- 影响行动池
- 影响事件权重
- 影响某些成功率
- 解锁专属因果线
- 改变结局标签

---

## 14. 地图系统在 v2 的定位

地图不再只是“去哪儿”的背景，而是人生阶段与压力分布的承载体。

## 14.1 地点类型建议沿用但重定义用途

现有地图设计可保留其地点分类，但要服务于：

- 当前阶段目标
- 主线压力推进
- 因果回收触发
- 道途培养

推荐四类 MVP 地点：

1. **安全区**：坊市、宗门、客栈、洞府
2. **资源区**：药田、矿洞、山林、湖泽
3. **危险区**：妖兽山脉、邪修遗迹、禁地外围
4. **事件区**：拍卖会、秘境入口、旧地重游、大劫战场

## 14.2 地图的 v2 新职责

地图应影响：

- 可选行动类型
- 行动成功率
- 因果回收触发条件
- 主线准备进度
- NPC 可遇见对象
- 阶段压力走向

例如：

- 某旧因果只能在旧地重游时触发
- 某主线准备必须在坊市收集情报、在山脉找材料、在宗门换资格

## 15. 事件系统：从“一次性事件”改为“阶段性事件链”

## 15.1 事件链原则

事件不能只是一次触发然后结束。

应改为：

- 有阶段
- 有倒计时
- 有失败分支
- 有后续回响
- 能和 NPC / 因果 / 主线交织

## 15.2 事件链字段建议

```ts
export type EventState = {
  id: string
  title: string
  category: 'main' | 'side' | 'npc' | 'world' | 'causality' | 'breakthrough'
  stage: number
  status: 'active' | 'resolved' | 'failed' | 'hidden' | 'cooldown'

  phaseHint?: string
  relatedNpcIds?: string[]
  relatedLocationIds?: string[]
  relatedCausalityIds?: string[]
  summary: string
  rewards?: string[]
  penalties?: string[]
  nextStageHint?: string
  flags: string[]
}
```

## 15.3 事件链类型建议

- 主线压力事件链
- 个人成长事件链
- NPC 关系事件链

## 20. 前端规则层与 LLM 的职责切分（v2 版）

- 状态真相
- 当前阶段
- 主线压力源
- 行动合法性
- 概率计算
- 数值结算
- 事件推进
- 因果记录与触发
- 存档
- 结局标签判断

## 20.2 LLM 负责的新增重点

相比旧版本，LLM 需要新增支持：

- 阶段感的当前局势描述

- 玩家是否死亡

- 事件是否满足前置条件
- 因果是否真正触发
- 存档真值

## 22. 回合生成逻辑（v2 版）

推荐顺序：

1. 推进月份
2. 推进主线压力
3. 推进事件链
4. 检查因果是否达到 tracking / returned 条件
5. 生成当前阶段目标提示
6. 根据地点、阶段、主线、关系、因果生成候选行动
7. 保证行动池中有：
   - 至少 1 个稳健行动
   - 至少 1 个成长行动
   - 至少 1 个与当前阶段/主线/因果相关的行动
8. AI 润色当前局势与风险提示

## 25. 给 AI 编码执行的实施顺序建议

- 倒计时展示

## 26. 成功判断标准

v2 成功的判断，不是文本更华丽，而是：

1. 玩家能明确说出“我现在这个阶段在干什么”
2. 玩家会因为倒计时而规划行动
3. 玩家会记住“我以前做过的事后来回来找我了”
4. 玩家会区分不同人生路线，而不是只看数值涨跌
5. 玩家即使失败，也会因为结局总结和局外解锁愿意重开

如果玩家仍然表现为：

- 只是一直读剧情
- 不在意时间和阶段
- 不关心主线压力
- 不记得旧行为后果
- 觉得 NPC 只是文本对象

那么说明 v2 仍未完成本次重构目标。
